# Think Tank - Teachers Aide

This is a well-architected and ambitious project. You are aiming to build a sophisticated, unified AI assistant that functions both as an intelligent coding partner within an IDE and as a personalized study aid. The foundation of this system is a robust Retrieval-Augmented Generation (RAG) pipeline, enhanced by a dynamic context orchestration system inspired by the Model Context Protocol (MCP).

Here is my high-level understanding of the project based on your detailed architecture and the provided documentation:

### Project Vision

The primary goal is to create a seamless, intelligent environment that accelerates learning and development. This involves ingesting, understanding, and synthesizing information from diverse sources—ranging from code repositories and technical documentation (like the `teachers_aide.md` and `foundry_palantir_project.md` concepts) to study materials (like `intellistudy.md` and `rag_pipeline.md`), and structured data models (like `data_models.txt`).

### Core Architecture: Modular RAG with Dynamic Context

The architecture you’ve designed is modular, emphasizing a local-first approach and a separation of concerns that allows for scalability and maintainability. It moves beyond basic RAG by incorporating dynamic context assembly.

#### 1. Ingestion and Indexing (Offline)

This stage involves processing diverse data sources (Local Docs, Code, PDFs, APIs) into a searchable format.

*   **Extraction & Cleaning:** Utilizing tools like Unstructured.io or PyMuPDF to convert raw files into clean text.
*   **Chunking & Embedding:** Splitting the text into semantically coherent chunks with overlap. I note the strategy to tailor chunk sizes differently for code (IDE use-case) versus long-form documents (study assistant). Embeddings are generated using Hugging Face or OpenAI models.
*   **Storage:** A dual-storage approach: a Vector Store (pgvector/Pinecone) for fast semantic search and a relational Metadata DB (Postgres) for structured data, configurations, and ontology.

#### 2. Retrieval (Online)

When a query or IDE event occurs, the system identifies the most relevant information.

*   **Query Embedding:** The input is vectorized using the same model as the ingestion pipeline.
*   **Vector Search:** The system performs a similarity search (e.g., cosine similarity, potentially hybrid) to find the top-K relevant chunks.
*   **Reranking:** An optional but valuable step to refine the relevance of the top results using a cross-encoder (e.g., ms-marco-MiniLM), ensuring the most pertinent information is prioritized.

#### 3. Context Orchestration (The "MCP-Inspired" Engine)

This is the differentiator in your architecture, heavily informed by the concepts in `mcp.md`. Instead of only using the retrieved documents, the system dynamically assembles a comprehensive context package.

*   **The Protocol:** A configuration file (`context_protocol.yaml`) defines the required context for different scenarios.
*   **Fetcher Plugins:** Custom Python modules that execute at query time to pull in dynamic or static data not stored in the vector index (e.g., the current file open in the IDE, recent build statuses, or structured data from the Metadata DB).
*   **Assembly and Templating:** The retrieved chunks and the fetched dynamic data are merged and injected into a specific Jinja2 prompt template before being sent to the LLM.

#### 4. LLM Layer and Presentation

*   **LLM Proxy:** A unified interface that abstracts the specific LLM being used (local HF models or OpenAI API), allowing for flexibility.
*   **Response Formatting:** Post-processing the LLM output to add citations, metadata, and structure the response.

### Client Interfaces and Operations

The intelligence generated by the backend will be consumed through various clients: a VSCode Plugin for the IDE assistant, a Web UI (Streamlit or custom) for the study assistant and dashboards, and a CLI for automation.

The entire system is designed with operational maturity in mind, incorporating schedulers (Celery/Airflow) for managing ingestion pipelines and observability tools (Prometheus/Grafana/ELK) for monitoring performance and health.

### Summary

I understand that you are building a local-first, highly contextual AI platform. The architecture effectively leverages the strengths of RAG for knowledge retrieval while incorporating a flexible, MCP-style orchestration layer to inject dynamic, real-time data, enabling hyper-personalized and accurate assistance for both coding and studying use cases.

This is a significant elevation of the project. Moving from plugins and basic UIs to a fully custom IDE with dedicated React frontends is ambitious, but it offers total control over the user experience and allows for the deep integration required to blend coding and studying seamlessly.

Here is a recommended strategy and a phased implementation plan to tackle this complexity.

### Strategy: Unified Platform, Modular Architecture

#### 1. The Application Shell: A Unified Desktop Experience

While the IDE and Study Assistant serve different functions, I strongly recommend building a **single, unified desktop application**.

*   **Why Unified?** The core strength of your vision lies in the integration. A unified application allows seamless context sharing (e.g., the study assistant referencing the code currently being edited) and deep linking between notes and code.
*   **Technology Choice:** Use **Electron** or **Tauri**.
    *   **Recommendation:** Start with **Electron**. It is the mature foundation of VS Code, offering the best compatibility and ecosystem for building a complex IDE.

#### 2. The IDE Core: Leverage Existing Power

Do not build a code editor from scratch.

*   **Editor Engine:** Integrate the **Monaco Editor** (the open-source core of VS Code).
*   **Standard Intelligence:** Leverage the **Language Server Protocol (LSP)**. Monaco can communicate with existing LSPs (e.g., Pyright for Python) for standard features like error checking and basic autocomplete. Your AI will augment this.

#### 3. Project Architecture: The Monorepo

A project of this scale requires a **Monorepo** structure (using tools like Nx or Turborepo) to manage complexity and share code effectively.

*   `@app/backend`: The FastAPI RAG pipeline, MCP orchestrator, and APIs.
*   `@app/desktop`: The Electron shell and main application routing.
*   `@app/feature-ide`: React components for the IDE (Monaco, file explorer, terminal).
*   `@app/feature-study`: React components for the Study Assistant (Chat, Notes, Flashcards).
*   `@app/shared-ui`: Common design system, components, and API hooks.

#### 4. Backend Strategy: The Headless "Brain"

The backend remains a decoupled FastAPI service. For a local-first approach, the Electron application can manage the lifecycle of the Python backend process (starting it when the app launches). The MCP Orchestrator is the critical bridge, designed to accept real-time context from the frontend.

### Overall Implementation Plan

We will follow a phased approach that establishes the intelligence foundation before building the complex user interfaces.

#### Phase 0: Foundational Setup (1-2 Weeks)

**Objective:** Establish the project structure and infrastructure.

1.  **Monorepo Setup:** Configure the monorepo using Nx or Turborepo.
2.  **Infrastructure:** Set up Docker Compose for PostgreSQL (with pgvector) and Redis (for task queues).
3.  **Backend Initialization:** Initialize the FastAPI project.
4.  **Data Modeling:** Define the core ontology and SQLModel definitions (based on `data_models.txt` and `intellistudy.md`).

#### Phase 1: RAG Backbone (The "Brain") (3-4 Weeks)

**Objective:** Implement the core intelligence engine and ingestion pipeline.

1.  **Ingestion Pipeline:** Implement the Extractor (Unstructured.io/PyMuPDF), Chunker, and Embedder.
2.  **Asynchronous Processing:** Set up Celery (using Redis as the broker) to handle ingestion tasks in the background.
3.  **Vector Storage and Retrieval:** Configure pgvector and implement the basic vector similarity search.
4.  **LLM Proxy:** Set up the connection to the LLM (local HF or OpenAI) and a basic `/query` endpoint.
    *   **Milestone:** The backend can ingest documents and answer basic contextual questions via API calls.

#### Phase 2: MCP Orchestration and Advanced RAG (3-4 Weeks)

**Objective:** Implement the dynamic context assembly architecture.

1.  **MCP Core Implementation:** Implement the parser for `context_protocol.yaml` and the core orchestration logic.
2.  **Basic Fetcher Plugins:** Develop initial plugins for static data (e.g., project metadata) and retrieved document chunks.
3.  **Prompt Templating:** Integrate Jinja2 and define base templates.
4.  **Reranking (Optional):** Integrate a cross-encoder (e.g., MiniLM) to improve retrieval relevance.
    *   **Milestone:** The `/query` endpoint now uses the MCP framework to assemble context dynamically before calling the LLM.

#### Phase 3: Desktop Shell and Study Assistant (4-6 Weeks)

**Objective:** Build the application container and the first functional module (Study Assistant is less complex than the IDE).

1.  **Electron Shell & React Setup:** Initialize the Electron application and the main React structure (layout, routing).
2.  **Document Management UI:** Implement interfaces for uploading study materials and monitoring ingestion status.
3.  **RAG Chat Interface:** Build the primary chat UI, connecting it to the `/query` endpoint, including streaming responses and citation display.
4.  **Note-Taking:** Integrate a rich-text editor (e.g., Tiptap or Slate.js) and implement CRUD operations for notes.
    *   **Milestone:** A functional desktop application where users can upload documents, chat with the RAG system, and take notes.

#### Phase 4: The Custom IDE Core (The "Workbench") (5-7 Weeks)

**Objective:** Integrate the core functionalities required for a code development environment.

1.  **IDE Layout:** Implement the file explorer sidebar, editor pane, and status bar.
2.  **Monaco Integration:** Embed the Monaco editor into the IDE module.
3.  **File System Access:** Implement secure communication between the Electron main process (which has OS access) and the React renderer to manage files.
4.  **Integrated Terminal:** Integrate a terminal emulator (using `xterm.js` and `node-pty`).
5.  **Basic LSP Support:** Configure Monaco to communicate with standard Language Servers (e.g., for Python or TypeScript).
    *   **Milestone:** The application functions as a basic code editor with standard language support and terminal access.

#### Phase 5: Deep AI Integration (The "Copilot") (6-8 Weeks)

**Objective:** Merge the RAG intelligence with the real-time IDE environment.

1.  **Context Transmission:** Implement the mechanism for the frontend to capture and send the current IDE state (active file content, cursor position, selected text) with AI requests.
2.  **IDE-Specific Fetcher Plugins:** Develop the crucial MCP fetchers on the backend to process this real-time IDE data.
3.  **Contextual AI Chat:** Implement an IDE sidebar chat that automatically utilizes the IDE context.
4.  **AI Actions:** Implement context menu actions in Monaco (e.g., Right-click -> "Explain Code", "Generate Tests", "Refactor").
5.  **Inline Code Completion ("Ghost Text"):** (Highly Complex) Implement the mechanism to stream LLM-generated code suggestions directly into the Monaco editor based on the full RAG context.
    *   **Milestone:** The IDE provides intelligent, context-aware assistance grounded in both the codebase and the study materials.

#### Phase 6: Operations and Polish (Ongoing)

**Objective:** Ensure stability, performance, and deployability.

1.  **Advanced Study Features:** Implement AI Flashcard generation and automated study scheduling.
2.  **Observability:** Implement the monitoring stack (Prometheus/Grafana) to track RAG performance, latency, and accuracy.
3.  **Packaging and Distribution:** Configure Electron builders to package the application for Windows, macOS, and Linux.
